Roadmap para migrar el ecommerce a Clean Architecture pura

Documento operativo para que otro agente ejecute el refactor sin improvisar. El objetivo es preservar comportamiento, seguridad y criterios de las specs mientras se corrige la dirección de dependencias, se reducen acoplamientos y se sacan reglas de negocio de HTTP/infraestructura.

Proyecto: Maison Elite Ecommerce Stack actual: Bun + Elysia + Drizzle + Eta No sustituir stack No reclamar producción sin evidencia

1. Objetivo

Reorganizar el código hacia Clean Architecture estricta, manteniendo el monolito modular y los bounded contexts existentes. La migración debe ser incremental, verificada y sin reescritura completa.

Resultado esperado: las reglas de negocio viven en domain y application, los detalles viven en infrastructure y interfaces, y las dependencias siempre apuntan hacia adentro.

2. No Objetivos

  • No introducir Next.js, React SPA, microservicios, colas externas, Redis ni otra base de datos.
  • No cambiar reglas de negocio mientras se refactoriza arquitectura, salvo bug confirmado por specs/tests.
  • No mover archivos masivamente sin tests de caracterización antes.
  • No eliminar módulos como crm o stylist sin decisión explícita del usuario.
  • No marcar specs como completadas por refactor interno sin evidencia funcional.

4. Reglas Arquitectónicas Obligatorias

Dirección de dependencias

  • domain no importa nada de application, infrastructure, interfaces ni web.
  • application no importa getDb, schema.ts, SDKs externos, Elysia, Eta ni process.env.
  • interfaces adapta HTTP, cookies, forms y templates; no consulta DB directamente.
  • infrastructure implementa puertos, repositorios, gateways, proveedores y detalles externos.

Casos de uso

  • Reciben dependencias por parámetro o factory del composition root.
  • Devuelven DTOs explícitos, no filas Drizzle crudas.
  • Coordinan transacciones mediante un puerto UnitOfWork, no mediante getDb().
  • No renderizan HTML ni conocen redirects.

Repositorios y gateways

  • Las interfaces de repositorio viven en application/ports.
  • Las implementaciones Drizzle viven en infrastructure.
  • Providers de pago, notificaciones, AI y config se consumen por puertos.
  • No usar instanceof de infraestructura dentro de application.

HTTP y presentación

  • Las rutas solo validan input HTTP, llaman casos de uso y renderizan respuesta.
  • Los handlers no calculan stock, totales, reglas de pago ni autorización de dominio.
  • HTML manual permitido solo para fragments simples; preferir templates/partials.
  • Errores se traducen en presenter/mapper, no dentro del caso de uso.

7. Fases del Roadmap

Fase 0 - Línea Base y Guardrails

Primero

Antes de mover arquitectura, congelar comportamiento observable.

  1. Ejecutar bun run typecheck y bun test.
  2. Ejecutar integración real con RUN_DB_INTEGRATION=1 bun test tests/integration/checkout.test.ts si hay DB disponible.
  3. Agregar tests de caracterización para checkout confirm, retry de pago, returns de WebPay/MercadoPago y wishlist si faltan.
  4. Crear una regla de revisión: ningún nuevo import de shared/infrastructure/db desde application o interfaces.

Criterio de salida: baseline verde o fallos documentados antes del refactor.

Fase 1 - Puertos Compartidos Mínimos

P0
  1. Crear src/shared/application/ports/unit-of-work.ts con una abstracción transaccional mínima.
  2. Crear src/shared/application/ports/logger.ts si el logger debe usarse fuera de web.
  3. Crear src/shared/application/ports/clock.ts y id-generator.ts solo si ayuda a testear flujos críticos; no sobrediseñar.
  4. Implementar adaptadores Drizzle en src/shared/infrastructure.

Criterio de salida: los nuevos puertos existen, tienen tests unitarios mínimos y no cambian comportamiento.

Fase 2 - Refactor de Checkout

P0
  1. Crear checkout/application/ports/repositories.ts para carrito, checkout session y snapshots.
  2. Crear puertos consumidos por checkout: InventoryReservationPort, PricingPort, CustomerVerificationPort, OrderWriterPort, PaymentInitiationPort, OutboxPort.
  3. Mover reglas puras a dominio: estado de checkout, expiración, validación de transición, cálculo de snapshot requerido.
  4. Extraer ConfirmCheckoutUseCase como clase o factory con dependencias explícitas.
  5. Dejar checkout/presentation/storefront-routes.ts solo con parseo HTTP, llamada a use case y rendering.

Criterio de salida: checkout/application ya no importa getDb, schema.ts, repositorios concretos ni providers de pago.

Fase 3 - Refactor de Pagos

P0
  1. Renombrar la abstracción a PaymentGateway o mantener PaymentProvider, pero mover factory concreta fuera de application.
  2. Crear payments/application/ports/payment-gateway.ts, payment-repository.ts y payment-finalizer.ts.
  3. Eliminar instanceof MercadoPagoProvider y instanceof WebPayProvider de application.
  4. Modelar operaciones específicas como métodos del gateway: fetchExternalStatus, commitReturn, parseWebhook.
  5. Mover lógica de return desde presentation/return-routes.ts a casos de uso como HandlePaymentReturnUseCase.

Criterio de salida: agregar o modificar provider no requiere tocar casos de uso centrales salvo contrato.

Fase 4 - Rutas sin DB Directa

P1
  1. En storefront.ts, mover PDP, sitemap y robots a casos de uso/presenters.
  2. En cuenta cliente, mover favoritos a customers/application/use-cases.
  3. En fulfillment storefront, mover autorización por token público a un caso de uso.
  4. Eliminar imports de getDb y schema.ts desde interfaces.

Criterio de salida: grep de src/modules/**/presentation/*.ts no encuentra imports de DB/schema.

Fase 5 - Catálogo y Búsqueda

Hecho
  1. Separar query model de búsqueda en catalog/application/ports/search-read-model.ts.
  2. Mover SQL Drizzle a catalog/infrastructure/drizzle-search-read-model.ts.
  3. Mantener DTOs de PLP/PDP independientes de filas Drizzle.
  4. Verificar filtros, sort y paginación contra subspec de catálogo.

Criterio de salida: catalog/application/search-use-cases.ts no importa Drizzle ni schema.

Evidencia 2026-06-12: bun run typecheck PASS; bun test PASS (346 pass, 35 skip, 0 fail). Grep de catalog/application/search-use-cases.ts sin imports de Drizzle/schema.

Fase 6 - Config, Logger y Notificaciones

Parcial
  1. Reemplazar accesos directos a process.env fuera de config.ts.
  2. Reemplazar console.* fuera del logger central.
  3. Convertir NotificationSender en gateway inyectado desde composition root.
  4. Aislar AI stylist detrás de puerto y feature flag validado por config.

Criterio de salida: grep de process.env solo queda en config, tests o adaptadores explícitamente autorizados.

Evidencia 2026-06-12: process.env solo queda en shared/infrastructure/config.ts; console.* solo queda en shared/infrastructure/logger/index.ts; NotificationSender se inyecta desde worker.ts. bun run typecheck PASS; bun test PASS (346 pass, 35 skip, 0 fail). Pendiente: aislar AI stylist detrás de puerto/feature flag validado por config.

Fase 7 - Schema y Repositorios por Contexto

P2
  1. No dividir schema.ts al principio; primero ocultarlo detrás de puertos.
  2. Cuando application/presentation ya no dependan de schema, evaluar partición por archivos: catalog.schema.ts, orders.schema.ts, payments.schema.ts.
  3. Mantener un barrel de schema para Drizzle si facilita migraciones.
  4. Evitar ciclos entre schemas con referencias cruzadas no controladas.

Criterio de salida: partición opcional completa sin cambios de migración inesperados.

9. Testing y Verificación

Cada fase debe mantener la suite verde y agregar tests de caracterización antes de tocar flujos críticos.

bun run typecheck
bun test
RUN_DB_INTEGRATION=1 bun test tests/integration/checkout.test.ts
bun run test:e2e
Advertencia: si no hay DB o navegador disponible, documentar el comando no ejecutado, la razón y el riesgo. No marcar producción ni spec PASS con evidencia parcial.

Cobertura mínima por fase

  • Checkout: doble confirmación, carrito vacío, stock insuficiente, precio cambiado, email no verificado, pago caído y retry.
  • Pagos: return manipulado, monto distinto, moneda distinta, webhook duplicado, provider caído y firma inválida.
  • Rutas: app.handle para PDP, PLP/search, cart, checkout, account favorites y returns.
  • Arquitectura: grep o test estático que falle ante nuevos imports de DB en application/presentation.

10. Checks Estáticos Sugeridos

Hasta que exista una herramienta formal de boundaries, usar estos comandos como smoke checks manuales:

rg "shared/infrastructure/db" src/modules/*/application src/modules/*/presentation
rg "from .*infrastructure" src/modules/*/application
rg "instanceof .*Provider" src/modules/payments
rg "process\.env" src
rg "console\." src

Un resultado no vacío no siempre es fallo automático, pero debe estar justificado y reducido por fase.

11. Composition Root

La creación de objetos concretos debe concentrarse en web/composition para que el resto del sistema dependa de contratos.

src/web/composition/
  checkout.ts
  payments.ts
  catalog.ts
  customers.ts
  shared.ts

Ejemplo de intención:

// Pseudocódigo
export function makeConfirmCheckoutUseCase() {
  const db = getDb();
  const uow = new DrizzleUnitOfWork(db);
  return new ConfirmCheckoutUseCase({
    unitOfWork: uow,
    checkoutRepo: new DrizzleCheckoutRepository(db),
    inventory: new DrizzleInventoryReservationGateway(db),
    pricing: makePricingService(),
    payments: makePaymentInitiationUseCase(),
    outbox: new DrizzleOutboxGateway(db),
  });
}

12. Reglas de Migración Incremental

  1. Crear puerto.
  2. Crear implementación adaptadora usando el código existente.
  3. Inyectar adaptador desde composition root.
  4. Mover una sola ruta o caso de uso.
  5. Ejecutar verificación.
  6. Eliminar import viejo.
  7. Repetir.
Regla de tamaño: no mezclar refactor de arquitectura con cambios de comportamiento en el mismo commit/tarea, salvo que el test demuestre un bug existente y el usuario lo apruebe.

13. Definition of Done

  • bun run typecheck pasa.
  • bun test pasa.
  • Tests de integración relevantes pasan o quedan documentados como no ejecutados con razón.
  • No hay nuevos imports de infraestructura desde domain o application.
  • No hay DB directa desde rutas refactorizadas.
  • Los flujos afectados mantienen evidencia en tests o checklist manual.
  • No se actualizan specs a PASS sin evidencia.

14. Instrucciones para el Agente Ejecutor

  1. Leer AGENTS.md, specs/spec-auditoria-final.md y los subspecs del área antes de editar.
  2. Empezar por Fase 0 y no saltar a particionar carpetas sin puertos y tests.
  3. Priorizar checkout y pagos antes que estética de carpetas.
  4. Hacer cambios pequeños y verificables.
  5. Reportar archivos tocados, specs afectadas, comandos ejecutados y riesgos restantes.
